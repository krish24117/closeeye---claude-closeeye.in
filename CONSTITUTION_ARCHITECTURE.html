Close Eye · Product Law

Architecture Constitution

v1.0 · FROZEN

Ratified & frozen 2026-07-17 · build against it, not around it

PreamblePurpose

Close Eye is not an AI chatbot. Close Eye is a Family Intelligence and Execution Platform.

Its purpose is to understand a family's context, make trustworthy decisions within defined policies, coordinate trusted execution, and continuously build family memory.

The LLM is one component of this system — not the system itself.

Non-negotiableConstitutional Principles

  1. The Family Graph is the source of truth.
  2. Retrieve before Reason.
  3. Safety before Intelligence.
  4. The LLM understands language; Close Eye makes decisions.
  5. Execution follows policy, not prompts.
  6. Every interaction becomes memory.
  7. No critical execution depends solely on an LLM.

Article IEnd-to-End Request Lifecycle

Every request follows exactly this lifecycle.

  ↓

  ↓

  ↓

  ↓

  ↓

  ↓

  ↓

No component may bypass this lifecycle without an explicit architectural decision.

◆ Review addition — sanctioned exits & the safety lane

Three movements are constitutional exceptions to the linear flow, named here so no one "corrects" them: a crisis exits at the Safety Gate (an ambulance never waits on the rest of the pipeline); an outside-scope request terminates at the Router (a denial needs no pricing or workflow); and an emergency the LLM surfaces indirectly re-enters the Safety Gate rather than proceeding. Safety is a gate and a destination.

◆ Review addition — the Response Composer

The lifecycle understood the family and acted, but never spoke back. The LLM's second, constrained role sits between Execution and Memory: it phrases the reply warmly, only from what the deterministic layers decided. It speaks; it does not decide.

Article IIResponsibilities of Every Component

Gateway

Receive every request.

Allowed

  • Authentication
  • Authorization
  • Request validation
  • Rate limiting

Forbidden

  • Business logic
  • Medical reasoning
  • Family reasoning

Safety Gate

Protect life and platform integrity — emergency detection, abuse detection, security validation, compliance checks. Must never depend on an LLM.

Context Retrieval

Collect everything required before reasoning: Family Graph, Timeline, Health, Finance, Legal, Preferences, Guardians, previous interactions.

Context Builder

Transforms retrieved data into structured context. It does not reason — it organizes. ◆ It is also the trust boundary for retrieved content: injection inside an uploaded document is data the Gateway never saw, so what is built into the prompt is checked here.

LLM · Language Understanding

Exists to understand human language.

May

  • Extract intent
  • Resolve ambiguity
  • Identify entities
  • Ask clarifying questions
  • Summarize · translate
  • Structure information

May not

  • Decide policy
  • Execute services
  • Override deterministic rules
  • Invent family facts
  • Become the system of record

Structured Intent

Normalized output from a closed set — the model chooses, it does not compose — with an explicit unsure → human.

{ "person":"Father", "domain":"Health",
  "intent":"Hospital Companion",
  "priority":"High", "confidence":0.95 }
◆ Review note — confidence must be calibrated

A model's self-reported confidence is not a probability — LLMs are miscalibrated. The threshold that routes unsure → human is set against a labeled eval, not the raw number, or the field is theatre.

Capability Router

Consumes structured intent only — no natural-language parsing here. Determines: Execute · Coordinate · Inform · Outside Scope. Inform = facts and who handles it; never diagnosis, legal opinion, or investment advice.

Decision Engine

Close Eye's core intellectual property. Business rules · trust policies · pricing · permissions · escalation · workflow selection.

Execution Orchestrator

Coordinates execution with Guardians · Hospitals · Doctors · Chartered Accountants · Lawyers · Partners · future agents.

Response Composer ◆

The LLM phrases the reply to the family — only from what the deterministic layers decided. It speaks; it does not decide.

Memory Engine

Updates Timeline · Family Graph · Preferences · future retrieval context. Every completed interaction becomes durable memory. ◆ The audit trail is a separate, immutable store — see the note under Article V.

Article IIIInterfaces Between Components

Each component communicates only through structured contracts — never through free-form prompts. Every interface has a versioned schema.

LLM                  → StructuredIntent

Decision Engine      → ExecutionPlan

Article IVThe LLM — Boundaries

The LLM may

  • Understand natural language
  • Extract structured intent
  • Summarize conversations
  • Generate follow-up questions
  • Explain concepts · translate
  • Produce empathetic responses

The LLM must not

  • Decide emergencies
  • Authorize payments
  • Dispatch Guardians
  • Override policies
  • Invent family information
  • Become the source of truth
  • Skip retrieval
  • Execute actions independently

If the LLM is unavailable, deterministic safety and essential platform functions must continue operating.

Article VWhat Must Remain Deterministic

These components are constitutional and must not rely solely on an LLM.

Safety

  • Stroke detection
  • Cardiac indicators
  • Suicide risk
  • Abuse indicators
  • Emergency escalation

Identity

  • User authentication
  • Family member resolution
  • Authorization
  • Access control

Policy

  • Business rules
  • Pricing
  • Membership validation
  • Permissions · compliance

Execution

  • Guardian dispatch
  • Appointment booking
  • Workflow state
  • Audit logging
  • External integrations

Memory

  • Timeline updates
  • Family Graph persistence
  • Event history
  • Audit trail
◆ Review addition — how determinism is proven, and audit vs memory

Deterministic components are governed by regression suites; the LLM by evals. That is why Close Eye is allowed to trust its own code over the model — the trust is earned by tests, not asserted. And the audit trail must be immutable while family memory is correctable (Article X) — opposite by nature, so they are two stores, never one.

Article VIArchitectural Invariants

True regardless of future technology.

  1. The Family Graph is the system of record.
  2. The LLM is replaceable; the platform is not.
  3. Reasoning always follows retrieval.
  4. Execution always follows policy.
  5. Safety never depends solely on model availability.
  6. Every action is auditable.
  7. Every interaction strengthens family memory.
  8. Connect, Ask, and future products consume the same platform services.

Article IXPlatform Independence

Close Eye shall remain independent of any single AI provider. Anthropic, OpenAI, Google, or future models are replaceable language-understanding engines. ◆ Because no business logic ever lives in the model and every contract is versioned, a provider can be swapped at the cost of re-validating the understanding layer against its eval — never the platform.

The Family Graph, Decision Engine, Execution Orchestrator, and Memory Engine constitute the enduring core. As models evolve, Close Eye's advantage remains its proprietary family intelligence, trust model, execution network, and accumulated memory — not dependence on a particular AI model.

◆ Refinement

"Replaceable," not "interchangeable." Swapping a model is real work — the structured-intent schema and the classifier eval are re-proven per provider. The point is that the cost is bounded to the language layer and never touches the platform. That distinction is what makes the article true rather than aspirational.

Article X ◆Memory Ownership & the Right to Correct

The family owns its graph. Memory is durable, but not permanent: it is correctable when the engine misunderstood, and erasable on the family's request. Every stored family fact carries its provenance — stated by the family, or inferred by the system — and an inference never hardens into a fact.

For a platform holding health, finance, and legal information for Indian families, this is not optional — DPDP compliance lives here. "Every interaction becomes memory" (Principle 6) is bounded by this article: the graph grows with consent and shrinks on request. The immutable audit trail (Article V) records that a correction or erasure happened, without retaining the erased content.

◆ Why this is an article, not a footnote

The constitution enshrined durable memory but not the family's right over it. Durable and permanent are not the same word — and for regulated PII, the difference is the law.

Article XIRatification & Amendment

This document is frozen at v1.0 as of 2026-07-17. From this point:

  • The Constitution is immutable except through explicit architectural review. Implementation documents evolve freely; the Constitution does not move to fit them.
  • Build against it, not around it. When code and Constitution disagree, the code is wrong until an amendment says otherwise.

An amendment is valid only when all four hold — anything less is drift:

  1. A written rationale for what changes and why.
  2. A version bump (v1.0 → v1.1 for a clarification, v2.0 for a principle change).
  3. A commit that touches only this document — never bundled with implementation.
  4. Founder sign-off.

Every amendment is itself an auditable action (Invariant 6). The freeze is not "never change" — it is "change deliberately, in the open, and never silently."
