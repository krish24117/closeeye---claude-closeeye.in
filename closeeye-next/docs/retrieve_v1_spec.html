Retrieve v1 — the keystone read seam.

The stage that turns Constitution Principles 1, 2 and 6 from words into behaviour: the Family Graph read, RLS-scoped, versioned, feeding the understanding path — so Close Eye never reasons from an empty prompt.

Implementation doc · evolves freely · NOT the frozen Constitution

ScopeWhat v1 is, and is not

Is: a single function retrieve(client, subjectId?) → RetrievedContext that reads loved_ones + family_ledger through the caller's RLS-scoped client, applies provenance and correction rules, attaches coverage, and returns one versioned contract. Wired so the family app — not only /space — reads through it. First consumer: Connect's understanding path.

Is not: the LLM understanding step (that comes after), new domain tables (Health/Finance/Legal packs are v2+), any write path (retrieve only reads), or a caching layer. One seam, done correctly.

FoundationWhat already exists — v1 builds on it, adds no storage

The schema is already right for this. family_ledger is append-only, RLS-scoped by family_user_id, immutable (no UPDATE/DELETE policy — corrections are new rows), and carries provenance. v1 reads it; it changes no table.

The contractRetrievedContext v1 — a versioned schema (Article III)

// Provenance is preserved, never flattened — Article X: an inference is
// never presented as a stated fact.
type Provenance = 'stated' | 'observed' | 'inferred'

interface LedgerItem { label: string; body: string; provenance: Provenance;
                       source: string; at: string }

interface RetrievedContext {
  schemaVersion: 'retrieve.v1'
  subject: { lovedOneId: string; name: string;
             relationship: string | null; city: string | null } | null
  stated:      LedgerItem[]   // family_fact (connect/space) — the family's words
  observed:    LedgerItem[]   // guardian_observation, visit_observation
  inferred:    LedgerItem[]   // ai_understanding — Connect's reading, flagged
  coverage: { presence: 'available'|'unavailable'|'unknown',
              financial: 'available'|'unavailable'|'unknown' }  // from service-region
  meta: { retrievedAt: string; complete: boolean }  // complete=false → a source failed
}

Corrections are applied before the arrays are built: for a given label, the most recent row wins, and a correction or a stated fact always outranks an inference (Article X — memory is correctable, and an inference never hardens into a fact). Superseded rows are dropped from the context, never shown as current.

The one invariant that ends the company if wrong

Cross-family isolation. retrieve() reads through the caller's authenticated, RLS-scoped client — never the service-role key. Family A's retrieve can never return family B's rows, and this is enforced at the database (the select using (family_user_id = auth.uid()) policy) AND proven by test G1 below. A read seam that could be pointed at another family's graph is not shippable at any quality of the rest.

BehaviourFail-safe, per Principle 7

  • Empty graph → empty context, never a guess. A family with no ledger yet returns stated: [], inferred: [] — the understanding path then works from the message alone. Never invented.
  • A source fails → complete: false, partial data, and the pipeline degrades to "understand from the message + route to a human." Retrieval never fabricates context to fill a gap.
  • No subject given (general question) → subject: null, family-level facts only. The caller decides whether to ask "who is this about?"
  • Coverage is computed from the subject's city via the existing service-region.ts — so retrieval already carries "can we act here" into the context, deterministically.

Acceptance gateWhat must pass before it ships

The gate is the spec. Retrieve v1 is done when every one of these is a green regression test.

  1. G1 · Cross-family isolation. Two families, two graphs. A's retrieve returns zero of B's rows, under every call shape. Any path using the service-role key for a family read fails the gate outright.
  2. G2 · Never invent. An empty or partial graph yields empty/partial context — never a fabricated fact, city, or relationship.
  3. G3 · Provenance integrity. A family_fact arrives as stated; an ai_understanding arrives as inferred. An inference is never emitted as stated.
  4. G4 · Correction supersedes. A correction row for a label wins over the fact it corrects; the old value is absent from the context (Article X).
  5. G5 · Fail-safe. A simulated source failure yields complete: false and a usable partial context — never an exception that blocks the family, never fabricated filler.
  6. G6 · Versioned & stable. schemaVersion: 'retrieve.v1' is always present; the shape is the contract downstream builds against.

Where it plugs inThe Constitution lifecycle

Retrieve v1 fills the stage between the Safety Gate and the Context Builder (Article I). Its output — RetrievedContext — is the contract the Context Builder assembles into a prompt, and the thing today's deterministic Connect engine will read first instead of working from the raw sentence alone. When the LLM understanding step lands later, it consumes the same contract. Nothing downstream is rebuilt; Retrieve is the missing input, not a new pipeline.

Two decisions I'd want confirmed before building:

1 · The correction/supersede rule. v1 proposes "latest row per label wins, stated/correction over inferred." If corrections should instead reference the exact row they correct (a corrects_id column), that's a small migration — worth deciding now, since it's the one place the schema might grow.

2 · First consumer. I propose wiring Retrieve into Connect's authed path first (where the graph and a real family exist), leaving the public /connect stranger flow deterministic-as-today. Confirm that's the right first surface.
