Architecture transformation plan · for approval · not built

India-first → global platform.
An extraction, not a rewrite.

The honest headline: ~60% of the spine already exists and was built to anticipate this — the frozen Constitution's layering, coverage as domain × mode, emergency as a country-extensible resource pack, and a deterministic engine that already needs no humans and runs on-device anywhere. What remains is real work — a region-config layer, extraction of hardcoded assumptions, full i18n, an API gateway, and the one genuinely hard infra problem: multi-region data residency.

The clarifying principle that resolves "Connect must work without Care": Connect gives emergency GUIDANCE (global, a config'd local number). Care gives physical RESPONSE (regional). 108 is configuration, never a Care dependency — the resource-pack pattern already proves this.

Deliverable 2 · 3 · 5Global platform + regional service + modular architecture

Connect (global core) — Family Intelligence, Memory, Understanding, Reports, Timeline, Document Vault, Emergency Guidance, Knowledge Graph, Notifications, Permissions. Depends on nothing regional and no human. Works in every country the day it deploys.

Care (regional plugin layer) — Presence Manager, Guardian Visits, Hospital Companion, Local Assistance, Physical Emergency Response, Financial, Insurance, Property. Each is a module that registers itself for the regions where it's live. The platform functions with zero modules present.

// lib/care/registry.ts — Care modules are plugins; Connect never imports them directly.
interface CareModule {
  id: 'presence'|'hospital'|'financial'|'insurance'|'property'|'community'
  availableIn(region: RegionCode): boolean
  offer(ctx: NextAction, region: Region): CareOffer | null   // null = not here → Connect shows nothing
}
// The Capability Router (Constitution Art. I) asks the registry, not a country if-block:
router.coordinate(intent)  →  careRegistry.forRegion(user.region).filter(m => m.offer(...))

This is the Constitution's Execution Orchestrator made regional: Guardian / Hospital / CA / Partner become registered Care modules. Adding a city is registering a module for a region — no core change, exactly the success criterion.

Deliverable 4 (keystone)The Region Configuration Layer

Extend the embryo in service-region.ts into the single source of regional truth. Every hardcoded assumption reads from the resolved Region. No business logic branches on country.

// lib/platform/regions.ts
type RegionCode = 'IN'|'GB'|'CA'|'AU'|'US'|'DE'|'JP'|'BR'|'ZA' | string
interface Region {
  code: RegionCode
  connect: true                      // ALWAYS — Connect is global, never disabled
  care: Partial<Record<CareModuleId, boolean>>  // which modules are live here
  locale: { languages: string[]; default: string; rtl: boolean;
            currency: string; measurement: 'metric'|'imperial' }
  emergency: { number: string; label: string }      // 108 | 999 | 911 | 112 …
  dataResidency: 'in'|'eu'|'us'|'apac'          // which cluster the family's data lives in
  phoneFormat: string; addressSchema: 'generic'|'us'|'in'|'jp'
}
// India = today's behaviour, expressed as config (zero behaviour change on day 0):
IN: { connect:true, care:{presence:true,financial:true}, emergency:{number:'108',…}, dataResidency:'in' }
GB: { connect:true, care:{}, emergency:{number:'999',…}, dataResidency:'eu' }   // Connect only
CA: { connect:true, care:{}, emergency:{number:'911',…}, dataResidency:'us' }

Resolution: regionFor(user) from the family's country (signup) → the loved-one's location for Care coverage. The existing coverageFor(domain, location) becomes coverageFor(region, module, location).

Deliverable 6 · 7Data model + database changes

  • profiles.region_code + loved_ones.region_code — set at signup, drives config resolution and residency. Migration: default 'IN' for all existing rows (no behaviour change).
  • family_ledger is already region-agnostic — no change. Good.
  • Consent records per regulation — extend consent.ts: consent(user, purpose, regulation, granted_at, region) so GDPR/CCPA/DPDP each have an auditable basis.
  • Data residency = the hard one. One Supabase project today. Options, cheapest → correct: (a) launch Connect-only regions on the same cluster with contractual SCCs (fast, legally thin); (b) regional Supabase projects keyed by dataResidency with a routing layer (correct, months). Recommend (a) for early Connect-only markets, (b) before EU volume.
  • Numbers/currency stored as minor-units + ISO currency code, never a ₹ string.

Deliverable 5 (API)API-first — one contract, many clients

Today: web calls Supabase directly (RLS) + scattered edge functions. For mobile / desktop / partners / marketplace to share one platform, introduce a versioned API gateway — the Constitution's structured contracts (Art. III) become the public API:

POST /v1/understand        → StructuredIntent          // Connect, global
GET  /v1/family/:id/graph  → RetrievedContext           // Retrieve, RLS-scoped
POST /v1/care/offers       → CareOffer[]  (region-gated) // Care, regional
POST /v1/emergency/guidance→ { number, steps }          // Connect; number from region config

Same auth (JWT), same RLS, same crisis floor in front. Partners consume /v1/care/* without touching Connect — the marketplace-readiness criterion (Deliverable 9) falls out for free.

Deliverable 8UI changes

  • Every tel:108 → tel:{region.emergency.number} with its label.
  • Care surfaces (Book a visit, Presence Manager) render only when region.care.presence — a UK family simply never sees them; Connect (understanding, memory, guidance) is unchanged.
  • Currency, dates, numbers → Intl.NumberFormat / Intl.DateTimeFormat with the locale — never a hand-formatted string.
  • RTL: logical CSS properties (margin-inline, padding-inline) + dir from locale. The scoped-CSS system makes this a contained sweep.

Deliverable 9 (loc)Localization plan — locale-aware, not translated strings

  • Framework: next-intl (App-Router native) + ICU message format (plurals, gender, interpolation the brand voice needs).
  • Extraction: the strings are inline JSX today — a real sweep. Start with /connect + Family v2 (the value surfaces), defer marketing.
  • Deterministic engine caveat: understand.ts reads English keywords. A non-English family typing Hindi/Spanish needs either per-language keyword packs (the engine already has romanised Hindi/Telugu in crisis.ts — the pattern exists) OR the LLM-understanding step (Constitution's target) which is language-agnostic. This is the strongest argument to prioritise the LLM understanding stage for global.
  • Locale-aware: Intl for numbers/dates/currency; libphonenumber for phones; measurement from region.

Deliverable 10 · 11Security + Privacy (GDPR · CCPA · DPDP)

  • Consent management: extend consent.ts into a per-regulation ledger; an explicit consent step before the first health interaction, regulation resolved from region.
  • Data export + deletion: the machinery exists (delete-account) — surface a self-serve export (JSON of the family's graph) and delete in /space profile. Constitution Article X made this law; this builds it.
  • Audit log (immutable) ≠ memory (correctable): Article V — two stores. A deletion records that it happened without retaining the erased content.
  • Residency: per Deliverable 6 — EU data in EU before EU volume.
  • Security unchanged and strong: RLS is structural, the crisis floor is shared and deterministic, Retrieve's isolation is by-construction. Global adds surface (more regions, more partners) → the API gateway is the single place to enforce it.

The three parts money and cleverness cannot shortcut

1 · Data residency — a real multi-region infra project (single Supabase today). 2 · Full i18n — every inline string, plus the deterministic engine's English keywords (which is why the LLM-understanding stage matters for global). 3 · The API gateway — the seam every future client depends on. Everything else is extraction into the region layer.

Deliverable 12 · 13 · 14Migration, rollback, roadmap

Config-driven means every phase defaults to today's India behaviour — so rollback is flipping a flag, not reverting code. Each phase ships to UAT, gated, before the next.

PHASE 0 · foundation (no behaviour change)

Region config layer + region_code columns

Build regions.ts; add region_code (default IN); route every hardcoded 108/₹/city READ through the resolved region. India behaves identically.

Rollback: the layer resolves to IN for everyone → byte-identical to today.

PHASE 1 · Connect-only region (proof)

Turn on ONE Connect-only market (e.g. GB)

Care flags empty; emergency number 999; Connect (understanding, memory, guidance) works; Care surfaces hidden. Proves the split end-to-end with real users.

Rollback: remove GB from the region table — no code change.

PHASE 2 · i18n + LLM understanding

next-intl + the language-agnostic understanding stage

Extract strings; wire the Constitution's LLM understanding (also unblocks non-English input). Locale-aware formatting everywhere.

Rollback: default locale en; the deterministic engine remains the fallback.

PHASE 3 · API gateway + Care registry

Versioned APIs; Care as registered plugins

Extract contracts into /v1/*; move Guardian/Hospital/Financial behind the Care registry. Marketplace-ready by construction.

Rollback: clients keep calling the current paths until cut over per-endpoint.

PHASE 4 · data residency

Regional clusters before EU/US volume

The irreversible one — stage carefully, dual-write, verify, cut over per residency zone. Do this LAST and only when a region's volume demands it.

Rollback: dual-write window means the origin cluster stays authoritative until verified.

RisksRanked

  • P0 · Emergency guidance correctness per country — a wrong emergency number is lethal. Every region's number is a reviewed, tested config entry, pinned like the crisis floor.
  • P0 · Data residency mis-step — EU data outside EU is a breach. Phase 4 is last and staged.
  • P1 · The deterministic engine is English — a non-English family gets degraded understanding until the LLM stage lands. Mitigate: launch Connect-only regions where English is common first (GB/CA/AU), then i18n before DE/JP/BR.
  • P1 · Care/Connect coupling leaks — any if (country === 'IN') that survives is a bug. Enforce with a lint rule banning country literals outside regions.ts.
  • P2 · Cost of LLM understanding at global scale — per-call cost; the deterministic engine stays the free first pass, LLM only when it can't resolve.
